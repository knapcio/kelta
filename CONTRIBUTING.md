# Contributing to Kelta

Kelta is an architectural prototype. Contributions should strengthen or test its central claim: UI work should be proportional to state deltas and actual DOM changes, not component-tree size.

## Before opening a pull request

1. Open an issue for semantic changes or new runtime concepts.
2. Keep the optimizable IR serializable and free of arbitrary callbacks.
3. Add a reference-semantics test before adding an optimization.
4. Run the complete check locally:

```sh
npm run check
```

No dependency installation is currently required beyond Node itself.

## Development workflow

```sh
npm test
npm run build
npm run serve
```

Tests use Node's built-in test runner. The generated `dist/plan.json` is useful when reviewing compiler changes, and `window.__KELTA__` exposes the running engine and its last transaction journal in the demo.

## Design constraints

- Runtime components and virtual DOM nodes are out of scope.
- Keys must be stable and serializable.
- Transactions must remain atomic and deterministic.
- A compiler optimization must preserve the reference semantics.
- Browser mutations must preserve native semantics, accessibility and node identity.
- Unsafe HTML and implicit server-to-client data movement are not acceptable shortcuts.

## Pull requests

Keep changes focused. Explain the affected IR/runtime invariant, include behavioral tests, and report the output of `npm run check`. Performance claims need a reproducible workload and must include initialization, update work, memory and generated bytes where relevant.
