01 / WHY KELTA

Stop rediscovering the interface.

Most UI runtimes ask application code to describe the screen again, then work out what changed. Kelta makes change the primary input.

A

COMMON COMPONENT MODEL

Recompute output

  1. State setter fires
  2. Component code runs again
  3. Output trees are compared
  4. DOM changes are discovered

Work is organized around ownership and rerender boundaries.

Δ

KELTA EXECUTION MODEL

Propagate a delta

  1. A typed transaction commits
  2. Changed facts enter the graph
  3. Affected queries emit deltas
  4. Known DOM slots are patched

Work is proportional to affected data and actual output changes.

01

Serializable by design

The optimizable program is data, not a forest of opaque closures. An AI, DSL or conventional authoring layer can emit the same plan.

02

Effects stay explicit

Transactions are synchronous, atomic and rollback-safe. Async work cannot quietly keep an inconsistent render open.

03

The platform stays intact

Kelta emits semantic HTML and performs direct DOM operations. It does not replace the browser with a canvas or a UI virtual machine.

03 / BENCHMARK CONTRACT

05 / RESEARCH STATUS

A working core, not a production promise.

Kelta is intentionally small enough to audit. The execution model is real and tested; the ecosystem and several hard semantics are still research work.

WORKS TODAY / 0.1

  • Typed cells and normalized keyed tables
  • Filtered and ordered collections
  • Incremental count and sum aggregates
  • Atomic transactions with rollback
  • SSR and browser resumption
  • Direct text, property, class and keyed DOM operations

NOT YET / DELIBERATE GAPS

  • Async resources and server actions
  • Joins, groups and nested collections
  • Cost-based query planning
  • Event-level code splitting and workers
  • Foreign DOM ownership boundaries
  • Production optimizer and compatibility guarantees
